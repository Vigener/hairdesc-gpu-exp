# GPUプログラミング学習・実験成果報告 (HAIRDESC)

## 1. 目的と概要
本報告は、GPUプログラミング（OpenACC）の入門およびマルチGPU化（MPI+OpenACC）の実践として、特性の異なる2つの計算問題（N体問題、3次元拡散方程式）を対象に行った実験の成果をまとめたものである。
さらに、Miyabi が持つ2種類の計算ノード（GPU搭載の Miyabi-G と CPU専用の Miyabi-C）を用いた「CPU 最高性能 vs GPU 最高性能」の定量比較（Phase 4）も実施した。
単一ノードでの絶対性能（FLOPS・メモリ帯域）の評価と、複数ノードでのスケーリング性能（強スケーリング・弱スケーリング）を計測し、CPU と GPU のアーキテクチャ上の性能差や、マルチノード化における通信最適化の効果を検証した。

---

## 2. 実験環境と使用ハードウェア

実験は東京大学・筑波大学の共同利用スパコン「Miyabi」の2種類のノードを使用した。

### A. ハードウェア仕様

| | **Miyabi-G（GPU搭載ノード）** | **Miyabi-C（CPU専用ノード）** |
|---|---|---|
| **搭載ユニット** | NVIDIA GH200 (Grace Hopper) | Intel Xeon CPU Max 9480 × 2ソケット |
| **GPU** | H100 (1ノードにつき1GPU) | なし |
| **CPU** | NVIDIA Grace (ARM Neoverse V2, 72コア) | 60コア × 2S = 最大120コア |
| **GPU FP64 理論ピーク** | **67 TFLOPS** | – |
| **CPU FP64 理論ピーク** | ~1.8 TFLOPS (Grace, SVE2) | **6.72 TFLOPS** (AVX-512, 2ソケット計) |
| **メモリ帯域 (HBM)** | **HBM3: 3.35 TB/s** (H100) | **HBM2e: ~1 TB/s** (Xeon MAX 内蔵) |
| **使用ジョブキュー** | `debug-g` / `regular-g` | `short-c` |

> **Xeon CPU Max の特徴**: 通常の Xeon（DDR5: 300〜400 GB/s 程度）と異なり、高帯域メモリ（HBM2e）をパッケージ内に内蔵している。一般的な CPU より帯域が広いが、H100 の HBM3 との差は約 3.5 倍ある。

### B. ターゲット問題

#### B-1. N体問題 (N-body)
- **特性**: 演算律速（$\mathcal{O}(N^2)$）。粒子間の万有引力を総当たりで計算。
- **データ型**: 倍精度浮動小数点数 (`double`, FP64)
- **実験パラメータ（Phase 1〜3 GPU実験）**:
  - 粒子数 $N = 65{,}536$、タイムステップ数 1,000 ステップ
  - 総演算量: $20 \text{ FLOP/pair} \times N^2 \times 1000 \approx 8.59 \times 10^{13}$ FLOP
- **実験パラメータ（Phase 4 CPU vs GPU 比較）**:
  - 粒子数 $N = 65{,}536$（同じ）、タイムステップ数 100 ステップ
  - 総演算量: $\approx 8.59 \times 10^{12}$ FLOP

#### B-2. 3次元拡散方程式 (Diffusion3D)
- **特性**: メモリ律速・通信律速（ステンシル計算）。格子状の空間で隣接点のデータを用いて更新。
- **データ型**: 単精度浮動小数点数 (`float`, FP32)
- **実験パラメータ（弱スケーリング）**:
  - 1プロセス（1 GPU）あたりのローカルグリッド: $1024^3$（約10億グリッド）
  - タイムステップ数: 100 ステップ
  - 総演算量（1ノード）: $13 \text{ FLOP/cell} \times 1024^3 \times 100 \approx 1.39 \times 10^{12}$ FLOP
  - 必須データ移動量（1ノード）: $8 \text{ B/cell} \times 1024^3 \times 100 \approx 800$ GB

## 3. 単一ノード (1 GPU) における絶対性能評価

単一ノードにおける実行時間から、GPUの実行性能（実効 FLOPS・実効メモリ帯域）を算出した。

| 問題 | 実行時間 | 実効演算性能 | 実効メモリ帯域 |
|---|---|---|---|
| **N-body** ($N=65536$, $S=1000$) | 63.46 秒 | **1.35 TFLOPS** | – (演算律速のため) |
| **Diffusion3D** ($1024^3$, $S=100$) | 0.918 秒 | 1.51 TFLOPS | **871 GB/s** |

### 【考察】理論ピーク性能との比較

**メモリ帯域（Diffusion3D）**:
- 実効帯域 871 GB/s は H100 の理論ピーク帯域 3,350 GB/s の **約26%** に相当する。
- Phase 4 の実測結果から Miyabi-C（Xeon MAX, HBM2e）の実効帯域は約 500 GB/s と推定されるため、GPU は CPU のおよそ **3〜4倍** の実効帯域を発揮している。
- ステンシル計算は隣接セルへの局所的なアクセスパターンが多く、キャッシュヒット率が高いため、理論帯域に対して比較的高い割合が引き出せている。

**演算性能（N-body）**:
- 倍精度で 1.35 TFLOPS を達成。H100 の理論ピーク 67 TFLOPS の **約2%** にとどまる。
- これは N-body の内ループに含まれる `sqrt()` がボトルネックとなっており、FMA の連続発行を前提とした理論ピーク性能とは条件が異なるためである。
- Phase 4 では同じ問題で CPU（Xeon MAX）が 171 GFLOPS（理論比 2.5%）を示しており、GPU と CPU の双方で同様の `sqrt()` ボトルネックが生じていることが確認された。

## 4. MPIによるマルチノード実行と通信最適化 (Phase 1〜3, Miyabi-G GPU)

複数のGPUで問題を分散して解くため、MPIを用いたマルチGPU化の実験を行った。

### A. N-body の強スケーリング (Strong Scaling)
問題サイズ（$N=65536$）を固定したままノード数を増やした。

| ノード数（GPU数） | 実行時間 (秒) | 速度向上率 | 実効総演算性能 (TFLOPS) |
|:---:|:---:|:---:|:---:|
| 1 | 63.46 | 1.00x | 1.35 |
| 2 | 32.40 | 1.96x | 2.65 |
| 4 | 16.50 | 3.84x | 5.20 |
| 8 | 8.50 | 7.46x | **10.10** |

**【結果と得られた知見】**
N体問題は「計算量 $O(N^2)$ に対して通信量 $O(N)$」の比率であるため、計算時間が通信時間を大きく上回る。そのため、Unified Memory を使ったシンプルな MPI 通信実装でも、8ノードで **7.46倍（10.1 TFLOPS）** という、理想の線形スケーリングに近い結果が得られた。

### B. Diffusion3D の弱スケーリング (Weak Scaling) と 通信最適化
1ノードあたりの問題サイズ（$1024^3$）を固定し、ノード数の増加に伴い全体サイズを拡張した。
ステンシル計算はメモリ・通信律速であるため、3段階の最適化フェーズを経て性能を検証した。

| ノード数 | Phase 1: Unified Memory | Phase 2: GPU-Aware MPI | Phase 3: 通信と計算の隠蔽 |
|:---:|:---:|:---:|:---:|
| 1 | 1.71 秒 | 0.918 秒 | 0.924 秒（ベースライン） |
| 2 | 1.82 秒 | 0.975 秒 | 0.951 秒 |
| 4 | 1.91 秒 | 1.046 秒 | 1.022 秒 |
| 8 | 2.09 秒 | 1.207 秒 | **1.027 秒** |

**【最適化の過程と得られた知見】**
1. **Phase 1（アンチスケーリング）**: Unified Memory 使用時は CPU-GPU 間の暗黙のページフォルトがノード間通信のたびに発生する。ノード数を増やすほど実行時間が伸び、スケーリングが悪化した。
2. **Phase 2（通信待ちの露呈）**: `host_data use_device` プラグマにより GPU 間の直接通信（GPU-Aware MPI）を実現した。ページフォルトが解消され実行時間は約半減（1ノード: 1.71秒 → 0.918秒）したが、同期通信 (`MPI_Send/Recv`) の待ち時間が蓄積し、8ノードでは 1.207秒 に微増した。プロファイラの分析から、GPU カーネルの計算時間自体は 1〜8ノード間でほぼ一定（0.74秒）であり、増加分は通信オーバーヘッドによるものと判明した。
3. **Phase 3（良好な弱スケーリングの達成）**: `MPI_Isend/Irecv` による非同期通信と OpenACC の `async` を組み合わせ、通信中に内部領域の計算を並行して進める実装に変更した。ネットワーク待ち時間を計算の裏側に隠蔽できた結果、8ノード（全体約128GBの領域）においても 1.027秒 と、1ノード時（0.924秒）に近い実行時間が得られ、良好な弱スケーリングを達成した。

## 5. CPU vs GPU 性能対照実験 (Phase 4: N-body)

Phase 1〜3 はすべて Miyabi-G の GPU 上での実験であった。
Phase 4 では **Miyabi-C（CPU専用ノード）** と **Miyabi-G（GPU搭載ノード）** の両方を用い、同一の N 体問題を同一条件で実行して CPU・GPU の性能を直接比較した。

- **CPU 版**: MPI + OpenMP ハイブリッド (`nbody_cpu_fast`, `mpiicpx -O3 -qopenmp -axSAPPHIRERAPIDS`)
- **GPU 版**: MPI + OpenACC (`nbody_gpu_fast`, `mpicxx -fast -acc=gpu -gpu=cc90`)
- **共通条件**: $N = 65{,}536$ 粒子、$\text{STEPS} = 100$、ノード数 1〜2

### Strong Scaling 結果

| ハード | ノード数 | 実行時間 (s) | 実効 GFLOPS | 対1ノード比 | スケーリング効率 |
|---|---|---|---|---|---|
| **CPU** (Xeon MAX, 112 OMP) | 1 | 50.10 | 171 | 1.00x | – |
| **CPU** (Xeon MAX, 112 OMP) | 2 | 30.85 | 278 | 1.62x | 81% |
| **GPU** (H100) | 1 | 6.43 | 1,336 | 1.00x | – |
| **GPU** (H100) | 2 | 3.31 | 2,595 | 1.94x | 97% |

### カタログスペックと実測値の比較

| | **CPU (Xeon MAX, 1ノード)** | **GPU (H100, 1ノード)** |
|---|---|---|
| 理論ピーク FP64 | 6,720 GFLOPS (6.72 TFLOPS) | 67,000 GFLOPS (67 TFLOPS) |
| **実測 GFLOPS** | **171 GFLOPS** | **1,336 GFLOPS** |
| **ピーク比効率** | **2.5%** | **2.0%** |
| 実測性能比（GPU/CPU） | – | **7.8倍** |

### 考察

**① GPU は CPU の約 8 倍高速（N-body, 1ノード比）**
1ノード同士の比較では GPU が CPU の 7.8 倍の GFLOPS を達成した。
理論ピーク比が 9.95 倍であるのに対して実測比が 7.8 倍にとどまっているのは、どちらも `sqrt()` がボトルネックとなっており、FMA の連続発行を前提とした理論ピーク性能とは乖離が生じているためである。両者とも実効ピーク効率は約 2〜2.5% と同水準であり、実効 GFLOPS の差は絶対的なメモリ帯域の差によるものといえる。

> **【補足】`sqrt()` がボトルネックになる理由**
> CPU・GPU ともに、`sqrt()` は通常の積和演算（FMA）とは別の**特殊関数ユニット（SFU）** で処理される。
> SFU のスループットは FMA ユニットの約 1/16〜1/20 しかなく（Xeon MAX: FMA は 16 FLOP/サイクル、`sqrt()` は ~0.8/サイクル）、
> しかもその結果が出るまで後続の乗算・加算は開始できない（データ依存）。
> N-body の内ループには粒子間距離を求める `sqrt()` が 1 回必ず含まれるため、FMA ユニットが空いていても `sqrt()` の完了を待つことになる。
> これは CPU・GPU を問わずハードウェア上の制約であり、ソフトウェアだけでは解消が難しい。
> 近似計算（`rsqrt()` + 精密化）や高速多重極展開法（FMM）への切り替えが本質的な対策となるが、
> 今回の実装ではシンプルな総当たり法かつ倍精度演算を採用しているため、両ハードウェアで同水準の低効率になっている。

**② N-body の CPU 性能は L3 キャッシュ帯域に律速されている**
N 体問題は「演算律速」と分類されることが多いが、算術強度が $20 \text{ FLOP} / 56 \text{ bytes} \approx 0.36 \text{ FLOP/byte}$ と低く、ルーフラインモデル上ではメモリ律速の領域に位置する。
1ステップあたりのメモリ読み出し量は $N^2 \times 56 \approx 240 \text{ GB/step}$ であり、Xeon MAX の L3 有効帯域（推定 ~500 GB/s）で割ると $\approx 48$ 秒（100ステップ）となり、実測 50.1 秒とよく一致する。GPU は HBM3 帯域（3.35 TB/s）で律速され、同様の計算で $\approx 7.2$ 秒と推定でき、実測 6.43 秒と整合する。
**帯域比（3.35 TB/s / ~0.5 TB/s ≈ 6.7 倍）が実測性能比（7.8 倍）とほぼ一致しており、この問題における GPU の性能優位の本質はメモリ帯域の差である。**

**③ GPU のスケーリング効率が CPU より高い（97% vs 81%）**
GPU（97%）の方が CPU（81%）より良好なスケーリング効率を示した。
CPU の効率低下は `MPI_Allgather` による同期通信オーバーヘッドが主因であり、GPU 版は GPU-Aware MPI で通信の待ち時間が相対的に小さくなっている。

## 6. 総括

本実験を通じて得られた主な知見を以下にまとめる。

**単一ノードの絶対性能**:
- OpenACC の基本的なプラグマ（`#pragma acc parallel loop`）を追加するだけで、Diffusion3D では実効帯域 871 GB/s、N-body では 1.35 TFLOPS の性能が得られた。
- ただし、理論ピーク比での効率は演算器構成や問題の算術強度によって大きく異なる。今回の条件ではいずれも理論ピークの 2〜26% 程度にとどまり、理論値をそのまま比較基準にすることは適切でない。

**CPU vs GPU の性能差（Phase 4）**:
- N-body（$N=65536$）において GPU（H100）は CPU（Xeon MAX, 112スレッド）の **約 8 倍** 高速であった。
- この性能差の本質は演算コア数の差ではなく、メモリ帯域の差（HBM3 vs L3 帯域、約 6.7 倍）であることが帯域律速モデルから確認できた。
- GPU は Strong Scaling において 97% の効率を示し、マルチノード構成でも性能が良好に維持された。

**マルチGPU 化と通信最適化（Phase 1〜3）**:
- 演算律速な N-body は通信量が少ないため、シンプルな MPI 実装で 8ノード 7.46 倍の線形スケーリングが得られた。
- メモリ律速な Diffusion3D は Unified Memory のままでは通信ごとのページフォルトでスケーリングが悪化した。GPU-Aware MPI と非同期通信（計算と通信のオーバーラップ）を組み合わせることで、8ノードに拡張しても 1ノード時とほぼ同等の実行時間（弱スケーリング）を達成できた。
