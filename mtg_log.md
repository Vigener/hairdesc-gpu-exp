# GPUプログラミング学習・実験成果まとめ (HAIRDESC)

## 1. 学習の目的と方針
HAIRDESCの学習資料に基づき、GPUプログラミング（OpenACC）およびマルチGPU化（MPI+OpenACC）の実践手法を習得する。
Miyabi-G (NVIDIA Grace CPU + H100 GPU / GH200) 環境において、プロファイリング手法を駆使しながら、ハードウェア性能を最大限に引き出す最適化プロセスを検証した。

### ターゲット問題
- **N体問題 (N-body)**: 演算律速の代表例 ($\mathcal{O}(N^2)$)。
- **3次元拡散方程式 (Diffusion3D)**: メモリ律速・通信律速の代表例（ステンシル計算）。

---

## 2. 実装・実験フェーズの推移と成果

### Phase 1: Managed Memory (Unified Memory) を用いた初期実装
- **目的**: 開発コストの低いUnified Memoryを利用した単純なMPI並列化の評価。
- **結果**:
  - **N体問題**: 計算量が通信量を圧倒するため、8ノードでほぼ理想的な線形スケーリング(Strong Scaling)を達成（1ノード63.4秒 → 8ノード8.5秒）。
  - **3次元拡散方程式**: GH200の超広帯域メモリにより計算が一瞬で終わる反面、データ通信時にCPU-GPU間のページフォルトが発生。ノード数を増やすほど遅くなる「アンチスケーリング」を観測。

### Phase 2: GPU-Aware MPI による直接通信
- **目的**: Phase 1のアンチスケーリングを克服するため、`host_data use_device` プラグマを用いてGPUデバイスメモリ間で直接MPI通信を実施。
- **結果 (Weak Scaling, 1ノードあたり1024^3グリッド)**:
  - ページフォルトオーバーヘッドが消滅し、1ノード時の実行時間が1.71秒から0.91秒へ半減。
  - プロファイラ (Nsight) の解析により、GPUのカーネル計算時間は1〜8ノードで完全に一定(0.74秒)であることが判明。計算性能は完璧にスケールしていることを確認。
  - 一方で、同期通信 (`MPI_Send/Recv`) の待ち時間が全体実行時間を微増させていることが明らかになった。

### Phase 3: 通信と計算のオーバーラップ（隠蔽）
- **目的**: Phase 2で顕在化した同期通信の待ち時間を解消するため、非同期通信と非同期計算を並行実行する。
- **実装**:
  - `MPI_Isend` / `MPI_Irecv` による非同期通信。
  - 通信に依存しない内部領域を OpenACC の `async` ストリームで先行して並行計算。
- **結果**:
  - 8ノードでの実行時間が 1.20秒 (Phase 2) から 1.02秒 に短縮。
  - 1ノード時 (0.92秒) との差はわずか 0.1秒。データサイズが8倍(約128GB)になっても、ネットワーク遅延を見事に計算の裏側に隠蔽することに成功。

---

## 3. 結論
一連の最適化ステップ（Unified Memory → GPU-Aware MPI → 通信と計算のオーバーラップ）を通じて、Miyabi-G (GH200) の性能を極限まで引き出すことに成功した。
OpenACCとMPIを用いたマルチGPUプログラミングにおいて、通信のボトルネックを段階的に解消し、理想的なスケーリングを達成する実践的な最適化手法を確立できた。
